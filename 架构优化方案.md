### 1.3 当前架构的 5 个痛点

```
痛点 1: 单文件膨胀
    3863 行在一个文件中 → 修改任何一处都要滚动数千行
    → 编辑器性能下降，diff/merge 冲突概率增加
    → 多人协作基本不可能

痛点 2: 零测试保护
    PackingEngine (837行装箱算法) 没有任何测试
    → 修改算法时只能靠肉眼验证 3D 结果
    → 回归风险极高，扼制重构意愿

痛点 3: CDN 依赖
    Three.js 通过 CDN 加载
    → 首次使用需要网络
    → CDN 宕机则应用不可用
    → 无法锁定版本（滚动发布风险）

痛点 4: 无构建管道
    代码格式/校验/压缩/打包全部手工
    → 没有 lint/format，代码风格不一致
    → 无法使用 ES modules、TypeScript 等现代特性
    → 生产环境没有代码压缩（3,863 行原始文件）

痛点 5: 存储上限
    localStorage 5MB 上限
    → 规格库 + 历史记录多了会溢出
    → 无法跨设备同步
```

## 二、架构目标

### 2.1 目标状态

```
目标: 从"单文件原型"进化为"可维护的工程化产品"

核心指标:
├── 可维护性 ↑ 代码行/文件 < 800 行
├── 可测试性 ↑ 核心算法测试覆盖率 > 90%
├── 可部署性 → 维持单文件部署能力（可选）
├── 离线能力 ↑ 首屏加载后完全离线可用
└── 可扩展性 ↑ 新增功能不修改已有模块
```

### 2.2 架构原则

| 原则 | 说明 |
|------|------|
| **渐进演进** | 不搞"大爆炸"重构，每次改动都可独立交付 |
| **可逆决策** | 优先选择容易撤销的中间步骤 |
| **物理隔离** | 模块必须拆分为独立文件，不能只靠注释 |
| **测试护城河** | 算法代码必须先有测试再重构 |
| **维持零部署** | 最终产物仍然是一个可双击打开的 HTML 文件 |

## 三、优化方案

### 3.1 阶段一：文件拆分（预计 2~3 天）

这是**最紧迫、ROI 最高**的改动。不改变任何逻辑，仅做物理分离。

#### 目标文件结构

```
F:\CodeBuddy_projects\box-calculator\
├── index.html              ← 入口文件，只加载模块
├── src/
│   ├── css/
│   │   └── style.css       ← 全部 CSS (290行)
│   ├── js/
│   │   ├── app-state.js    ← AppState (157行)
│   │   ├── packing-engine.js ← PackingEngine (837行)
│   │   ├── storage-manager.js ← StorageManager (47行)
│   │   ├── visualizer-3d.js  ← Visualizer3D (639行)
│   │   ├── ui-renderer.js    ← UIRenderer (544行)
│   │   └── app.js           ← App + 初始化 (1002行)
│   └── lib/                ← 未来: 第三方库本地化
│       └── three.min.js
├── dist/                   ← 构建产物
│   └── index.html          ← 合并后的单文件（可部署）
```

#### 迁移策略

```
步骤 1: 创建 src/ 目录，原样复制每个模块到独立文件
    → 每个文件顶部加上 // ===== ModuleName =====
    → 保持 IIFE 模式不变
    → 验证：模块加载后全局可访问

步骤 2: index.html 改为通过 <script src=""> 加载
    → 加载顺序: style.css → app-state.js → packing-engine.js → 
      storage-manager.js → visualizer-3d.js → ui-renderer.js → app.js
    → 验证：页面功能完全正常

步骤 3: 将 CSS 移到 src/css/style.css，index.html 只保留 <link>

步骤 4: 引入 Vite 作为构建工具
    → 开发模式: npm run dev (带 HMR)
    → 构建模式: npm run build → 输出 dist/index.html (合并单文件)
```

#### Vite 配置（关键决策）

```javascript
// vite.config.js
export default {
  root: './src',
  build: {
    outDir: '../dist',
    rollupOptions: {
      input: './src/index.html',
      output: {
        inlineDynamicImports: true,  // 合并为单文件
        // 输出一个包含所有 CSS/JS 的独立 HTML
      }
    },
    cssCodeSplit: false,  // CSS 内联到 HTML
    assetsInlineLimit: 100000,  // Three.js 内联
  }
}
```

**为什么选 Vite 而不是 esbuild/parcel？**

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Vite ✅** | 开箱即用 HMR，开发体验好，插件生态丰富 | 需要 Node.js 环境 |
| esbuild | 极致的构建速度 | 需要手动配置插件链 |
| parcel | 零配置 | 对单文件输出支持不 |
| Rollup | 灵活可定制 | 配置复杂，学习成本高 |

#### HTML 入口重构

```html
<!-- 开发模式: 加载源文件 -->
<link rel="stylesheet" href="css/style.css">
<script type="module" src="js/app-state.js"></script>
<script type="module" src="js/packing-engine.js"></script>
<!-- ... -->
<script type="module" src="js/app.js"></script>

<!-- 构建后: 全部内联到 dist/index.html -->
<!-- 双击即可打开，维持零部署特性 -->
```

### 3.3 阶段三：三.js 本地化（1 天）

#### 问题

当前 Three.js 通过 CDN 加载：

```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r132/three.min.js"></script>
```

依赖 CDN 意味着：
- 首次加载必须联网
- CDN 域名被墙/宕机则应用不可用
- 无法锁定补丁版本

#### 方案

```
步骤 1: npm install three
步骤 2: import * as THREE from 'three'
步骤 3: Vite 构建时自动将 Three.js 打包到最终 HTML
步骤 4: 总文件体积: 当前 ~600KB (Three.js r132 min) + 自定义代码 ~50KB ≈ 650KB
        → 仍在合理范围内（一次页面加载）
```

**备选方案**: 使用 Three.js CDN + Service Worker 预缓存

```
开发期简单，但首次加载体验不如内联方案
推荐结合阶段四的 PWA 一起考虑:
  - 内联 Three.js → 首屏即离线
  - PWA 缓存未来版本更新
```

#### Service Worker 策略

```javascript
// sw.js
const CACHE_NAME = 'box-calculator-v1';
const ASSETS = [
  '/',
  '/index.html',  // 单文件已包含所有资源
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then(cache => cache.addAll(ASSETS))
  );
});

self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      // 缓存优先，网络兜底
      return cached || fetch(event.request).then(response => {
        return caches.open(CACHE_NAME).then(cache => {
          cache.put(event.request, response.clone());
          return response;
        });
      });
    })
  );
});
```

### 3.5 阶段五：重量约束（2~3 天）

这是 V2_PLAN.md 中规划的 **P0 功能**，改动量最小但用户价值最大。

#### 数据模型扩展

```
纸箱新增:
  weight:      number (kg)   ← 单件重量
  crushResist: number (kg)   ← 可选，抗压强度

木箱新增:
  maxLoad:     number (kg)   ← 最大承重
```

#### 算法改动

```
原有流程:
  纸箱列表 → 几何排列 → 输出结果

加入重量后:
  纸箱列表 → 按重量降序排列 → 几何排列 → 检查总重 →
  ├─ 未超重 → 输出结果
  └─ 超重   → 减少纸箱数量 → 重新排列 → 输出结果 + 警告
```

#### 实现要点

```javascript
// 改动范围估算

// 1. AppState 新增字段: 约 5 行
state.boxes[].weight = 0;
state.boxes[].crushResist = 0;
state.crate.maxLoad = 0;

// 2. UI 输入层: 约 15 行
// ... 重量输入框、承重输入框

// 3. PackingEngine 重量检查: 约 30 行
function addWeightCheck(crate, placedBoxes) {
  const totalWeight = placedBoxes.reduce((sum, b) => sum + b.weight, 0);
  if (totalWeight > crate.maxLoad) {
    return { overload: true, totalWeight, maxLoad: crate.maxLoad };
  }
  return { overload: false, totalWeight };
}

// 4. UIRenderer 展示层: 约 20 行
// 利用率旁增加"载重率"

// 总计: 约 70 行新增代码
```

#### 改动范围总结

| 文件 | 改动类型 | 预估行数 |
|------|---------|:-------:|
| app-state.js | 新增字段 | ~5 行 |
| packing-engine.js | 新增重量检查函数 | ~30 行 |
| ui-renderer.js | 新增输入框 + 结果展示 | ~35 行 |

### 3.6 阶段六：TypeScript 迁移（可选，2~3 天）

#### 价值分析

| 维度 | 收益 | 成本 |
|------|------|------|
| 类型安全 | 装箱算法参数复杂，TS 能捕获大部分类型错误 | 需要学习 TS |
| 开发体验 | VSCode 智能提示、重构工具 | 额外编译步骤 |
| 新人上手 | 类型即文档 | 需要了解接口定义 |

#### 渐进策略

```
步骤 1: 将 .js 重命名为 .ts，any 类型先放着
步骤 2: 优先为 PackingEngine 的接口定义类型
   type BoxInput = { l: number; w: number; h: number; qty: number; ... }
   type CrateInput = { l: number; w: number; h: number; maxLoad?: number }
   type PackingResult = { usedCount: number; utilization: number; ... }
步骤 3: 逐步为其余模块补类型
```

**建议**: 当前阶段 **不推荐** TS 迁移。Gain 不足以 justify 成本。等模块拆分 + 测试体系完善后再考虑。

## 五、附录

### 5.1 模块接口契约

当前 IIFE 模块存在隐式依赖（通过闭包访问全局变量）。拆分后需要明确定义接口契约：

```javascript
// app-state.js — 导出全局 AppState
window.AppState = (function() {
  // ... 当前代码 ...
  return { getState, setState, addBox, removeBox, ... };
})();

// packing-engine.js — 依赖 AppState
window.PackingEngine = (function() {
  // ... 从 AppState.getState() 读取输入 ...
  return { calculateSingle, calculateMixed, calculateReverse, ... };
})();
```

### 5.2 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:----:|:----:|---------|
| 拆分后 HTML 加载顺序出错 | 中 | 严重 | 加载顺序用 <script> 顺序保证 |
| Vite 构建配置导致功能异常 | 低 | 严重 | 拆分阶段不引入构建，构建是独立步骤 |
| 测试覆盖不全导致回归 | 中 | 中 | 优先为已有算法写测试，不改逻辑 |
| 团队不熟悉 Vite | 低 | 低 | 配置简单，复制即用 |

### 5.3 收益预估

| 阶段 | 直接收益 | 量化指标 |
|------|---------|---------|
| 阶段一 | 单文件 3863 行 → 7 个文件，每个 < 900 行 | 文件内代码行数减少 4 倍 |
| 阶段二 | 算法回归有保障 | 测试覆盖率目标 > 90% |
| 阶段三 | 离线可用 | 无网络也能正常工作 |
| 阶段四 | 类 Native App 体验 | 可添加到桌面 |
| 阶段五 | 新功能: 重量约束 | 补全 V2 P0 功能，缩小与 CX7 差距 |

---

> **总结**: 当前项目处于"原型可用"到"工程化产品"的转型临界点。最迫切的改动是**文件拆分 + 测试体系**（阶段一+二），投入 2 周即可显著提升可维护性和质量基线。重量约束（阶段五）ROI 最高，可在拆分后进行，快速补齐功能短板。
